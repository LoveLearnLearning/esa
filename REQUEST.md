## 二、目标架构（五层系统）

系统从底到顶分五层，每层依赖下一层。

```
第5层  应用界面     学生端 + 老师端 (Flutter)
第4层  推理引擎     推题逻辑 + 讲解深度调整 + 复习提醒
第3层  用户模型     每个人对每个知识点的掌握度 (0-100%)
第2层  知识图谱     学科知识点 + 依赖关系网
第1层  数据层       培养方案 / 课程大纲 / 历年试卷 / 教材 PDF
```

## 三、分模块需求

### 模块 A：聊天 REST API（最优先，阻塞项）

**现状**：[chat.py](file:///d:/4github/esa/backend/core/web/routers/chat.py) 完全为空，[webAPI.py](file:///d:/4github/esa/backend/core/web/webAPI.py) 只注册了 auth 路由。

**需求**：实现 [API.md](file:///d:/4github/esa/API.md) 中规划的全部 6 个接口，并将 Agent 集成到 Web API。

| 接口 | 方法 | 功能 |
|------|------|------|
| `/conversations` | GET | 获取当前用户对话列表 |
| `/conversations` | POST | 新建对话 |
| `/conversations/{id}` | PATCH | 重命名对话 |
| `/conversations/{id}` | DELETE | 删除对话 |
| `/conversations/{id}/messages` | GET | 获取历史消息 |
| `/conversations/{id}/messages` | POST | 发送消息 → 调用 Agent.run() → 返回结果 |

**具体要求**：
- 所有接口需认证，只能访问属于当前用户的对话
- `POST .../messages` 内部调用 `Agent.run()`，传入历史消息和用户输入
- 对话不存在或不属于当前用户返回 404
- 后续应支持流式返回（SSE）

### 模块 B：RAG 知识库检索

**现状**：完全没有实现。

**需求**：构建学科知识库的向量检索系统，让 AI 回答前先查知识库，保证内容有据可查。

| 子项 | 说明 |
|------|------|
| 向量数据库 | ChromaDB 或 FAISS |
| Embedding 模型 | 开源 embedding 模型（如 BGE、text2vec），本地运行 |
| 知识来源 | 教材 PDF、课程大纲、历年试卷、学术论文 |
| 处理流程 | PDF 提取文本 → 分块(chunk) → Embedding → 存入向量库 |
| 检索方式 | 用户问题 → Embedding → 向量相似度检索 → 返回 top-k 片段 |
| 来源标注 | 每个检索片段必须保留来源文档名、章节、页码 |
| 索引管理 | 支持按学科/课程分类，支持增量更新 |

**工具注册**：注册一个 `retrieve_knowledge` 工具，Agent 在回答知识性问题时自动调用。


---

### 模块 C：知识图谱

**现状**：完全没有实现。

**需求**：构建学科知识点的依赖关系网，支撑后续的用户模型和推题逻辑。

| 子项 | 说明 |
|------|------|
| 数据结构 | 知识点节点（id, name, course, weight） + 依赖边（prerequisite_of） |
| 存储 | SQLite 或 PostgreSQL，独立表 |
| 构建方式 | 第一阶段手工整理 5-8 门核心课；第二阶段用 LLM 辅助从课程大纲提取 |
| 依赖关系 | 如"链表 → 指针"，"二叉树遍历 → 递归" |
| 考试权重 | 每个知识点标注在该课程期末考试中的权重占比 |
| 学科范围 | 数学（高数/线代/概率论）、计算机（数据结构/OS/计网）、医学（根据教材确定） |

### 模块 D：用户模型（掌握度追踪）

**现状**：当前 CoreMemory 只能存键值对，没有结构化的掌握度数据。

**需求**：为每个用户维护一份知识点掌握度档案，随答题动态更新，随长时间未练衰减。

| 子项 | 说明 |
|------|------|
| 数据结构 | `(user_id, knowledge_point_id, mastery_level, practice_count, last_practiced_at)` |
| 存储 | SQLite 独立表 |
| 更新逻辑 | 答对 → 掌握度上升（越练增幅越小）；答错 → 适度下降 |
| 衰减逻辑 | 超过 N 天未练习该知识点 → 掌握度逐渐衰减 |
| 范围限制 | 10%–95%，不归零不到 100% |
| 优先级打分 | 掌握度低(50%) + 考试权重高(30%) + 距期末近(20%) → 推题优先级 |


---

### 模块 E：推理引擎（推题 & 自适应讲解）

**现状**：完全没有实现。

**需求**：综合掌握度、考试权重、教学进度，决定推什么题、怎么讲。

| 子项 | 说明 |
|------|------|
| 推题逻辑 | 遍历当前教学进度范围内的知识点 → 计算优先级得分 → 取最高 |
| 讲解深度 | 三档：初学（讲概念+例子）、有基础（讲重点+易错）、已熟练（讲进阶+辨析） |
| 前置知识追溯 | 如果某知识点薄弱 → 检查其依赖的前置知识点 → 优先补前置 |
| 复习提醒 | 长期未练的知识点 → 主动提醒用户复习 |

**注册工具**：注册 `recommend_practice` 工具（返回推荐练习的知识点列表）、`get_mastery_report` 工具（返回掌握度报告）。

**现状**：[build_prompt.py](file:///d:/4github/esa/backend/core/message/build_prompt.py) 只有通用角色设定。

**需求**：构建学科感知的 System Prompt，注入用户模型数据和 RAG 结果。

| 子项 | 说明 |

| 教学进度注入 | 当前教学周、课程进度 |
| 讲解规则 | 根据用户掌握的档位，自动调整回答深度 |
| 来源标注规则 | 要求 AI 在回答末尾附参考来源 |
| AI 标识 | 要求输出带"AI 生成"标识（赛题合规要求） |

### 模块 H：多模态支持

**现状**：完全不支持。

**需求**：赛题明确要求支持文本、图文、表格等多模态交互。

| 场景 | 说明 |
|------|------|
| 拍照上传手写作业 | 数学：识别并批改手写作业 |
| 截图识别公式 | 数学：截图 → LaTeX → 求解 |
| 报错截图识别 | 计算机：报错截图 → 定位问题 |
| 检验报告识别 | 医学：检验单 OCR → 解读指标 |
| 公式/图表渲染 | 输出端渲染数学公式和图表 |

**技术路线**：
- 上传图片 → OCR/MLLM 提取信息 → 后续处理
- 输出用 LaTeX/MathJax 渲染数学公式
- 上传文件用 PDF 解析工具

## 五、技术栈建议

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 大模型底座 | 讯飞星火 / Qwen3.5 开源模型 | 比赛推荐 + 本地可部署 |
| 向量数据库 | ChromaDB | 轻量，嵌入式，无需独立服务 |
| Embedding | BGE-small-zh / text2vec | 中文效果好，本地可跑 |
| 知识图谱存储 | SQLite 独立表 | 与现有数据层一致，简单够用 |
| 后端框架 | FastAPI（已有） | 无需变动 |
| 前端 | Flutter（已有） | 无需变动 |
| 部署 | Docker Compose | 一键启动，评委可复现 |
| 微调 | LoRA + vLLM adapter API | 热切换，毫秒级切换 |

## 七、约束与注意事项

1. **来源可追溯**是赛题硬性要求，每条涉及权威信息的回答必须标注来源
2. **医学场景**信息错误会直接影响评审结果，需重点验证
3. **AI 生成标识**是合规要求，不可省略
4. LoRA 训完后**不要合并进基座**，保持独立 adapter 文件
5. Demo 需**8 月初就上线**，给用户至少一个月的试用时间
6. **知识数据的权威性**优先于数量——先做好 5-8 门核心课
